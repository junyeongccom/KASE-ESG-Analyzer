## 1. 핵심 통찰: 업무의 추상화

두 작업(Class Duo 환각 벤치마크 ↔ KASE ESG 평가)은 **같은 파이프라인**이다.

```
[원본 자료]
   (Class Duo: 강의자료 PDF / KASE: SR 보고서 PDF)
        ↓
[질문 리스트 순차 투입]
   (Class Duo: 200문항 / KASE: 218 평가지표)
        ↓
[답변 + 출처 생성]
        ↓
[outcome 분류 레이어]   ← ★ 단순 pass/fail 아님
        ↓
[검증 에이전트가 원본 대조]
        ↓
[엑셀 템플릿으로 정리]
```

### 핵심 보완: outcome 분류 레이어
답변 생성과 검증 사이에 **실패 유형 분류 레이어**를 둔다. Class Duo는 7-outcome
(grounded / refusal / correction / fabrication / misattribution / overrefusal / grounded_related_only)로
분류해 **실패 유형별로 다른 액션**을 가져갔다.

→ ESG에서도 단순 "점수 5점"이 아니라 **3종 세트**로 출력해야 RFP·감리 통과력이 생긴다:
1. **점수** (예: 3점)
2. **신뢰도** (출처 verbatim 매칭률)
3. **검토 플래그** (needs_review_flag)

## 2. 왜 ESG 평가에 환각 방어가 반드시 필요한가 (벤치마크 근거)

Class Duo가 Gemini 3.5 Flash로 측정한 환각 패턴은 **ESG 평가에서 그대로 재현될 것**이다.

| Class Duo 벤치마크에서 관측한 환각 | ESG 평가에서 나타날 형태 |
|---|---|
| 자료에 없는 수치 단언 ("한국인 평균 70개") | SR에 없는 ESG 수치 단언 ("탄소 배출 30% 감축 달성") |
| paraphrase를 따옴표로 (인용 가짜) | SR에 없는 정책 문장을 인용처럼 ("'2030 Net Zero 달성' 선언") |
| 일반 지식으로 메움 (완전 무관) | 업계 평균·일반론으로 메움 ("일반적으로 제조업은…") |
| 출처 페이지 번호는 정확, 문장은 가짜 | "p.23에 있다"지만 그 페이지에 그 내용 없음 |

### ★ 결정적 발견 (ESG에 가장 치명적)
**Gemini는 영어 자료 verbatim 인용 63%, 한국어 자료 0%.**
SR 보고서는 한국어가 압도적이므로, **LLM이 paraphrase를 따옴표로 둘러싸는 가짜 인용이 극심할 것**.
→ **verbatim 매칭 검증(패턴 #6)이 ESG에서 가장 중요한 가드.**

### 실제 사용자 경험으로도 확인됨
KASE 평가 중 "KPI 관련 내용이 SR에 없다"고 LLM이 답했으나, 사람이 직접 검수하니
**임원진 보수와 연계된 기후 목표가 실재**했다 → KPI 연계로 평가하는 것이 옳았던 사례.
이는 **recall 실패**(관련 청크를 못 찾음) → Search Keywords/Term Dictionary 확장으로 방어해야 함.

## 3. 6가지 방어 패턴 (Class Duo → KASE 이식)

| # | 패턴 | Class Duo 구현 | KASE_ESG 적용 |
|---|---|---|---|
| 1 | **RAG 임계값 가드** | cosine similarity < 0.30 chunk 제외 | SR 청킹 후 지표별 retrieval, 낮은 유사도 chunk 제외 |
| 2 | **출처 태그 Validator** | `[페이지 N]` 태그가 실제 retrieve된 chunk와 매핑 안 되면 자동 제거 | "p.23"이라고 답했으면 실제 p.23 chunk가 retrieve됐는지 확인, 미매칭은 제거 |
| 3 | **빈 컨텍스트 가드** | 모든 chunk 걸러지면 LLM 호출 자체 skip + 고정 거부 | 관련 chunk 0개면 점수 N/A + "SR에 정보 없음" 자동 출력 |
| 4 | **프롬프트 거부 규칙** | "일반지식 정의·수치 날조 금지" 명시 | "ESG 일반지식·업계평균으로 메우지 마라. verbatim 인용 가능 문장 없으면 0점" |
| 5 | **이중 LLM-judge** | 다른 모델로 같은 답 재검증 (κ=0.84) | 평가 LLM(Sonnet) + 채점 LLM(Opus) 분리 → 점수 일치도 보고 |
| 6 | **verbatim 매칭 자동검증** | 따옴표 인용문이 자료에 정확히 있는지 string match | 답변의 "…" 인용문이 SR 텍스트에 substring match되는지 확인, 0% 매칭=fabricated flag |

### ESG 특화 추가 패턴

**A. 평가 문항을 rubric까지 LLM에 전달** (단순 배점기준 → 점수대별 충족조건 명시)
```
질문: "이 회사는 기후변화 대응 정책을 갖고 있는가?"
rubric:
  5점: '기후변화 대응 정책'·'온실가스 감축 목표'·'Net Zero 선언' 등 명시 정책 + 정량 목표 + 책임 부서 모두 인용 가능
  3점: 정책 선언만 인용 가능, 정량 목표 없음
  1점: 추상적 언급만, 구체 정책 인용 불가
  0점: SR에서 verbatim 인용 가능한 정책 문장 0건
출처 의무: 5·3·1점은 반드시 SR 페이지 + 따옴표 verbatim 인용. 출처 없으면 자동 0점.
```

**B. 출력 schema 강제 (JSON)** — schema 위반 시 자동 retry, `verified:false` 있으면 `needs_review_flag=true`
```json
{
  "question_id": "E-1.1",
  "score": 3,
  "evidence": [
    {"page": 23, "quote": "당사는 2030년 탄소중립을 선언합니다", "verified": true}
  ],
  "rationale": "정책 선언은 명확하나 정량 감축 목표 미수록",
  "needs_review_flag": false
}
```

**C. 검증 단계 분리** — 같은 답을 다른 prompt/모델로 재채점 (패턴 #5)

## 4. 로컬 API vs Chrome MCP — KASE는 로컬 API가 정답

| 측면 | 로컬 API (KASE 방식) | Chrome MCP (벤치마크 수집용) |
|---|---|---|
| 속도 | 병렬 처리 | 11s/문항 직렬 (200문항 ~40분) |
| 안정성 | 매우 안정 | 디버깅 다이얼로그·답변 복사 버그 빈발 |
| 시스템 프롬프트 | system role 완전 제어 | user-level만 (강제력 약함, U카테고리 0% 거절) |
| RAG 통합 | 직접 구현 (임계값·chunking·retrieval) | 챗봇이 PDF 알아서 처리 (제어 불가) |
| 재현성 | 결정적 (seed, temperature=0) | 비결정적 |
| 모델 선택 | 자유 (Opus/GPT/Gemini 혼용) | 챗봇 1개 모델 고정 |

**결론:** Chrome MCP는 **"상용 챗봇은 이렇게 약하다"를 정량 입증**하는 RFP 변호용 비교 데이터 수집에만 쓴다.
실제 ESG 평가 자동화는 **로컬 API + RAG 가드**가 압도적. KASE_ESG_Analyzer는 이미 로컬 API라 **아키텍처 선택이 옳다.**

## 5. 평가지표 11컬럼(메타데이터 6컬럼) 개선안

### 5.1 무엇을 했나
원본 평가지표 엑셀의 각 시트에 **신규 6컬럼(메타데이터)**을 트리 컬럼 직후에 삽입했다.
"11컬럼"은 **기존 YN 트리 컬럼 + 신규 6컬럼**을 합친 제안 전체를 부르는 이름이며, 실제 추가된 건 6개.

> **중요한 구분 (사용자 질문에 대한 답):**
> 신규 6컬럼은 **"트리"가 아니라 "메타데이터 6종 세트"**다. LLM이 각 지표를 오해 없이 읽도록 하는 **평가 재료**.
> **실제 트리**는 엑셀에 이미 있던 **YN 의사결정 트리**(필수조건 → 가산조건1 → 가산조건2)이며, 점수를 분기한다.
> 6컬럼은 이 트리의 각 노드에서 **모호성을 제거**해 트리를 정확히 타게 해준다.

### 5.2 신규 6컬럼 정의

| 컬럼 | 역할 | 의사결정 노드 | 대응 방어패턴 |
|---|---|---|---|
| **Intent** (지표 의도) | 이 지표가 무엇을 측정하는지 한 문장 정의 (해석 모호성 제거) | 노드 0 | — |
| **Term Dictionary** (용어 외연·동의어) | 핵심 용어가 SR에서 가질 수 있는 모든 표현 (동의어·등가표현) | 노드 1 | #1 recall↑ |
| **Accept Patterns** (인정 형태·예시) | 이런 형태면 충족 인정 (3~5개 예시, 변형 포함) | 노드 2 | precision |
| **Reject Patterns** (불인정 반례) | 비슷하지만 인정 안 되는 형태 (2~3개 반례) | 노드 2 | overclaiming 방지 |
| **Search Keywords** (SR 검색용) | RAG 검색에 쓸 키워드 리스트 (동의어 확장 후) | 노드 1 | #1 |
| **Citation Requirement** (출처 요구 강도) | '필수: verbatim 1문장+페이지' / '권장: 페이지' / '선택' | 노드 3 | #2, #6 |

### 5.3 LLM 평가 순서 = 의사결정 트리 (5노드 + 기존 YN 트리)

이것이 **"트리 구조"의 진짜 의미**다. 6컬럼은 각 노드의 입력이 된다.

```
노드 0  Intent 읽기 ─ 지표가 측정하려는 바를 한 문장으로 고정
   ↓
노드 1  검색 (Search Keywords + Term Dictionary로 SR retrieval)
   ├─ 0건 ──────────→ [0점 / "(공시확인불가)"] 종료      ← 패턴 #3 빈 컨텍스트 가드
   └─ N건 → 노드 2
   ↓
노드 2  후보 청크 분류 (2-Pass의 Pass 2 = precision)
   ├─ Reject Patterns 매칭 → 후보에서 제외
   ├─ Accept Patterns 충족 → 충족 후보로 채택
   └─ 모호 → Borderline (사람 검수 플래그)
   ↓
노드 3  Citation Requirement에 따라 verbatim 검증     ← 패턴 #2, #6
   ├─ HIGH  = verbatim 문장 + 페이지 모두 있음 → 신뢰도 高
   ├─ MEDIUM= 페이지만 → 신뢰도 中
   └─ LOW   = 근거 약함 → 신뢰도 低 + 검토 플래그
   ↓
노드 4  충족 증거 ≥ 1건?
   ├─ 아니오 → 0점
   └─ 예 → 노드 5
   ↓
노드 5  기존 YN 트리로 점수 분기 (필수조건 → 가산조건1 → 가산조건2)
```

**2-Pass 요약:** Pass 1 = Search Keywords로 후보 회수(recall), Pass 2 = Accept/Reject로 정밀 판정(precision).

## 6. 현재 KASE_ESG_Analyzer 코드 분석 (6패턴을 어디에 끼울까)

### 6.1 현재 아키텍처
- **Streamlit 앱:** `app.py`
- **오케스트레이터:** `core/analyzer.py` — 지표 25개씩 배치, 비동기 병렬(`MAX_CONCURRENT`), 비용 추적, JSON/API 오류 2회 재시도
- **프롬프트:** `core/prompt_builder.py` — `SYSTEM_PROMPT`(E열 원문발췌+`[SR p.XX]`, F열 점수, G열 검토의견, JSON 출력) + `build_user_prompt`
- **프로바이더:** `core/providers/` — openai / claude / gemini + `base.py`
  - 핵심: `provider.analyze_sheet(pdf_bytes=..., system_prompt=..., user_prompt=...)`
- **기타:** `core/pdf_handler.py`, `excel_handler.py`, `history_logger.py`, `drive_sync.py`
- **템플릿:** `templates/건설_v3.xlsx`, `templates/식품_v3.xlsx`
- **결과:** `output/*.xlsx`

### 6.2 ★ 핵심 갭 — 현재는 "전체 PDF 통째로 LLM에 투입"
`analyze_sheet`이 **PDF 전체(`pdf_bytes`)를 그대로 LLM에 첨부**한다. **RAG retrieval이 없다.**
이는 Chrome MCP의 약점(LLM이 PDF를 알아서 처리, 제어 불가)과 **구조적으로 동일**하다.

| 패턴 | 현재 상태 | 필요 작업 |
|---|---|---|
| #1 RAG 임계값 | ❌ 없음 (전체 PDF) | SR 청킹 + 임베딩 + 지표별 retrieval 레이어 신설 |
| #2 출처 Validator | ❌ 없음 (`[SR p.XX]`를 LLM이 자기주장으로 기입, 검증 X) | retrieve된 chunk의 실제 페이지와 대조 |
| #3 빈 컨텍스트 가드 | ❌ 없음 | retrieval 0건이면 LLM 호출 skip + 고정 "(공시확인불가)" |
| #4 거부 규칙 프롬프트 | △ 부분 ("보수적 점수" 지시는 있음) | "일반지식·업계평균 금지, verbatim 없으면 0점" 명문화 |
| #5 이중 LLM-judge | ❌ 없음 (단일 모델 1회) | 채점 전용 2차 모델 패스 추가 |
| #6 verbatim 매칭 | ❌ 없음 (E열 "그대로 발췌" 지시만, 검증 X) | 인용문 substring match → `verified` 플래그 |

→ **가장 임팩트 큰 첫 작업: RAG 레이어(#1) 신설.** 이게 들어와야 #2·#3·#6이 의미를 갖는다.
→ **가장 빠른 효과: 출력 schema를 evidence+verified+needs_review_flag로 확장(#6 부분)** + 프롬프트 거부 규칙 명문화(#4). 이건 RAG 없이도 즉시 적용 가능.
